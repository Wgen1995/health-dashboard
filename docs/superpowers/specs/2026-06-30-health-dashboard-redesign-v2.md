## 1. 问题陈述

现有 skill 存在 4 个问题：

| 编号 | 问题 | 根因 |
|------|------|------|
| P1 | 柱状图标签被截断 | Plotly 没设布局约束 |
| P2 | 分析文字与图表数据不一致 | AI 独立写文字，不读脚本输出 |
| P3 | 表格数值与图表数值不一致 | 两条计算路径不同步 |
| P4 | 大数据量数据丢失 | 无校验 + 会话压缩截断 |

用户环境约束（之前已确认）：
- Windows，无 Python，非技术用户
- 不希望看到程序代码文件
- 数据准确性是红线，数据不丢失是红线，质量第一
- **需要 LLM 语义分析**——AI 做分析，不是脚本算好了喂给 AI

## 2. 解决方案

采用方案 A：**单 SKILL.md + references/ 纯文本规范**。

借鉴 sourcecpt/gencpt 的设计哲学：references 模式（纯文本规范）、智能派发策略、防会话压缩三层兜底。不照搬多子 Phase 结构（流程不够复杂），但借鉴其大数据处理哲学。

### 目录结构

```
health-dashboard/
├── SKILL.md                      ← 主文件，全流程指令，全中文
└── references/
    ├── echarts-config.md         ← ECharts 防截断配置规范
    ├── aggregate-json-schema.md  ← 聚合 JSON 结构定义
    └── result-excel-structure.md ← 结果 Excel sheet 结构定义
```

### 删除的文件

现有 reference/ 下的代码骨架文件（.py/.js/.ps1/.html/.json）全部删除，转为 .md 纯文本规范。

### 解决 4 个问题的机制

| 问题 | 机制 |
|------|------|
| P1 标签截断 | ECharts 替代 Plotly + 防截断配置规范（references/echarts-config.md） |
| P2 文字与图表不同步 | AI 读聚合数据后做语义分析写文字，图表也引用同一数据源 |
| P3 表格与图表不一致 | 图表/表格/Excel 都从同一聚合数据派生，禁止硬编码数字 |
| P4 大数据丢失 | 分批处理策略 + checksum 校验断言 + 数据落地文件 + 防会话压缩三层兜底 |

### 分工（之前已确认，此处明确）

| 环节 | 谁做 | 为什么 |
|------|------|--------|
| Excel → 可读格式 | AI 用 opencode 原生工具（Bash/Read），按实际环境选可用方式 | 二进制转换，不是分析 |
| 基础计数（groupby 计数、均值、标准差等机械统计） | AI 用 opencode 原生工具，按实际环境选可用方式 | 大数据量必须先聚合缩小体积，这是机械操作不是分析 |
| **语义分析（发现规律、识别异常、交叉洞察、趋势判断、写态势感知）** | **AI（LLM 语义分析）** | 这是 AI 的核心能力，工具做不到 |
| 生成 HTML 看板 | AI 用 opencode 原生 Write 工具直接写 | AI 写 HTML 文件，内嵌 ECharts |
| 生成结果 Excel | AI 用 opencode 原生工具，按实际环境选可用方式 | 写 .xlsx 需要 OS 能力 |
| 生成 Markdown 报告 | AI 用 opencode 原生 Write 工具直接写 | 纯文本 |

**AI 用 opencode 原生工具直接操作，不生成临时脚本程序。** 和 sourceCPT 一样的哲学。

### 命令执行灵活度分级（借鉴 sourceCPT）

SKILL.md 中的命令示例按分级灵活度控制——不写死操作方式，AI 按实际环境自适应：

| 命令场景 | 灵活度 | 允许 AI 调整 | 禁红线 |
|---------|-------|-------------|-------|
| 平台检测 | **LLM 自适应** | 先检测 bash/PowerShell/其它，选对应风格 | 不许硬死单一方式 |
| Excel→可读格式 | **半开** | 有 Python 用 pandas，有 Node 用 xlsx，有 PowerShell 用 ZIP 解压，有别的方式也行 | 不许跳过格式转换直接读 .xlsx 二进制 |
| 基础计数聚合 | **半开** | 用什么工具聚合都行（pandas groupby / PowerShell Measure-Object / Node reduce / SQL / 其它） | 不许 AI 自己手工算（必须用工具算保证准确） |
| 聚合数据写盘 | **死命令不调** | 必须写盘 | 不许靠 AI 上下文维持，必须落盘 |
| checksum 校验 | **死命令不调** | 必须校验 | 不许跳过校验直接生成看板 |
| AI 语义分析 | **LLM 决策为主** | AI 按实际数据自适应发现洞察 | 不许自创数字，必须引用聚合数据中的精确值 |
| HTML 生成 | **LLM 决策为主** | AI 按视图清单自适应生成 | 必须遵循 ECharts 防截断配置规范 |

**设计哲学**：纯工程操作（写盘/校验）给死命令；环境适配型（Excel 读取/聚合）给半开；分析决策（语义分析/HTML 生成）LLM 自主。参考 sourceCPT 的"命令执行灵活度分级"。

## 6. 大数据处理设计（借鉴 sourceCPT 哲学）

### 6.1 设计哲学

借鉴 sourceCPT 的两个核心哲学：
- **"内存不可信，磁盘可信"** — 所有中间结果写盘，不靠 LLM 上下文维持
- **智能派发** — 按数据规模自动选策略，不搞一刀切

### 6.2 分批处理策略（借鉴 sourceCPT 智能派发）

临时脚本启动后先统计总行数，按规模自动选策略：

| 总行数 | 策略 | 说明 |
|--------|------|------|
| ≤10万行 | `single-batch` | 全量读入一次聚合（当前设计） |
| 10万-100万行 | `file-batched` | 按文件分批，每批聚合后写 partial_N.json，最后合并所有 partial 为 aggregate.json |
| >100万行 | `chunk-batched` | 每个文件内部也分块读取（流式聚合），不一次性加载整个文件到内存 |

**file-batched 策略流程**：
```
扫描所有文件 → 统计总行数 → 确定策略为 file-batched
    │
    ▼ 按文件分批（每批 ≤50 个文件）
    │
    ▼ 每批：读取 → 聚合 → 写 partial_N.json 到磁盘
    │
    ▼ 所有批次完成 → 读取所有 partial_N.json → 合并为 aggregate.json
    │
    ▼ checksum 校验：sum(各 partial 求和) === total_records
```

**chunk-batched 策略流程**：
```
扫描所有文件 → 统计总行数 → 确定策略为 chunk-batched
    │
    ▼ 每个文件：分块读取（每块 ≤5000 行），流式聚合
    │   ├─ 读块1 → 更新该文件的 groupby 计数
    │   ├─ 读块2 → 累加到该文件的 groupby 计数
    │   └─ ... 直到文件读完 → 写该文件的 partial.json
    │
    ▼ 所有文件完成 → 合并所有 partial → aggregate.json
    │
    ▼ checksum 校验
```

### 6.3 防会话压缩三层兜底（借鉴 sourceCPT SRC_ACCESS §5）

| 层 | 机制 | 实现 |
|---|------|------|
| 1 | 每批 partial 聚合立即写盘 | partial_N.json 每批完成即落盘，不靠 LLM 上下文维持 |
| 2 | 合并阶段读磁盘，不靠会话记忆 | 合并所有 partial_N.json 时从磁盘读取，不依赖 LLM 记住"上一批算了什么" |
| 3 | checksum 最终校验 | sum(各 partial 的 total_records) === meta.total_records，不通过则报错重跑 |

### 6.4 LLM 语义分析 + 不碰原始数据

- LLM 永远不接触原始 Excel 行数据
- PowerShell 在引擎内部完成所有读取+基础计数+聚合
- PowerShell 输出聚合数据（各维度 Top5、分布、趋势、异常率等机械统计结果）
- **AI 读聚合数据后做语义分析**：
  - 发现跨维度规律（如"北京朝阳区高风险占比异常高，可能与该地区饮食结构有关"）
  - 识别需优先干预的异常指标
  - 判断趋势是否逆转、是否有季节性模式
  - 生成干预建议（P0/P1/P2 分级）
  - **这些洞察是 AI 语义理解的产物，不是脚本预计算的**
- AI 引用聚合数据中的精确值写文字，不做算术，不自创数字
- 即使几百个文件，LLM 上下文负担不变（只读聚合结果，不读原始行）

### 6.5 大文件预警

| 单文件大小 | 处理 |
|-----------|------|
| <20MB | 正常处理 |
| 20-100MB | 提示"数据量较大，处理需数秒" |
| >100MB | 用 chunk-batched 策略，分块读取该文件 |
